## 1. Архітектурні Догми v2 "Journal Core"

- **Журнал — це єдина істина. Все інше — кеш.**\
  Канонічний стан простору (Space) є сумою append-only журналів усіх
  авторизованих пристроїв та агентів. Будь-який документ `.md` на диску, індекс
  у SQLite, векторний embedding чи візуальне граф-представлення — це
  детермінована проекція (View), яку можна повністю видалити та безболісно
  перезібрати з журналів з нуля.

- **Ідентичність незмінна. Семантика плинна.**\
  Унікальний ідентифікатор ноди `node_id` народжується один раз на базі першої
  події створення. Людські назви, категорії, папки, вектори та FQDN — це лише
  плинні метадані (аліаси), які змінюються івентами і резолвляться через індекс.
  Це забезпечує конвергенцію з моделлю ідентичності складних живих систем.

- **Конвергенція без координації (Zero-Coordination Sync).**\
  Жодна операція в системі не потребує того, щоб два пристрої домовлялися в
  реальному часі. Порядок подій встановлюється через HLC, злиття — через
  математичну логіку конфліктно-вільних типів даних (CRDT). Транспортний шар
  може губити, дублювати або міняти місцями файли подій — стан системи
  гарантовано збігається.

- **Нульова довіра до транспортного каналу (Zero-Trust Transport).**\
  Хмарний провайдер (Google Drive, OneDrive, iCloud) бачить лише зашифровані
  бінарні сегменти з непрозорими іменами. Хмара є суто сліпим кур'єром, який не
  здатний прочитати жодного слова з корпоративних баз знань.

- **ШІ — повноправний громадянин системи, а не просто інструмент.**\
  Кожен автономний ШІ-агент або нічний демон має свій приватний ключ і веде
  власний журнал подій. Він діє у межах чітко окреслених капабіліті, пропонує
  зміни за межами своїх повноважень і несе повну репутаційну відповідальність за
  свої дії.

## 2. Ідентифікація та Адресація Простору

### 2.1. Формування стабільного Node ID

Унікальний ідентифікатор ноди `node_id` обчислюється один раз при її створенні
за допомогою криптографічної хеш-функції BLAKE3 від байтів її Genesis-події:

$$\text{node\_id} = \text{Base32}(\text{BLAKE3}(\text{genesis\_event\_bytes}))[0 \dots 25]$$

Genesis-івент містить у собі ULID (який поєднує мілісекундний таймстамп та 80
біт криптографічної ентропії), ідентифікатор автора та початковий тип сутності.
Будь-яке подальше перейменування, переміщення по теках або зміна семантичної
категорії є звичайним івентом мутації метаданих і не впливає на `node_id`.

### 2.2. Віртуальні FQDN як мутабельні аліаси

Для зручності користувача та ШІ система підтримує роботу з людськими доменними
іменами:

```
task.design-homepage.marketing.privilege  --->  node_id: 7K3M9H2B...
```

Мапінг FQDN -> `node_id` розраховується динамічно локальним резолвером на основі
подій встановлення аліасів (`ALIAS_SET`). Історія змін зберігається, що дозволяє
резолверу за запитом повертати дерево версій або стан аліасу на будь-який момент
часу.

### 2.3. Безпечний Path-Based Веб-Резолвінг

Для виключення уразливостей CORS та інфраструктурного оверхеду з генерацією
Wildcard SSL-сертифікатів, посилання для зовнішнього світу проксіюються через
єдиний шлюз на Cloudflare Worker:

[https://minddoc.md/r/space_id/fqdn_or_node_id](https://minddoc.md/r/[space_id]/[fqdn_or_node_id])

**Протокол публікації (Publish Protocol):**

1. Хмарний воркер за замовчуванням не має доступу до простору користувача і
   повертає 404.
2. Користувач емітить івент `PUBLISH_SNAPSHOT`, який локально збирає підграф,
   шифрує його одноразовим симетричним ключем і виштовхує у KV-сховище воркера.
3. Посилання генерується з ключем дешифрування у URL-фрагменті (hash):
   `https://minddoc.md/r/[space_id]/[node_id]#key=[aes_key]`.
4. Браузер клієнта отримує зашифровані дані, а JS-код у браузері дешифрує їх на
   стороні клієнта за допомогою ключа з фрагмента URL. Ключ ніколи не
   передається на сервер Cloudflare.

## 3. Специфікація Журнальної Моделі та Гібридні Логічні Годинники (HLC)

### 3.1. Математична специфікація HLC

Для встановлення тотального порядку подій у децентралізованому середовищі без
використання центрального сервера часу кожен пристрій реалізує Гібридні Логічні
Годинники (HLC).

HLC-таймстамп $T_i$ на пристрої $i$ складається з трьох компонентів:
$(l_i, c_i, \text{node\_id}_i)$, де $l_i$ — фізичний час (мілісекунди), $c_i$ —
логічний лічильник для подій у мебах однієї мілісекунди.

#### Алгоритм 1: Генерація локальної події на пристрої $i$

При створенні нової події зчитується поточний фізичний час системи
$\text{pt}_i$:

1. Обчислюємо нове логічне значення фізичного компонента $l'_i$:
   $$l'_i = \max(l_i, \text{pt}_i)$$
2. Обчислюємо значення лічильника $c'_i$:
   $$c'_i = \begin{cases} c_i + 1, & \text{якщо } l'_i = l_i \\ 0, & \text{в іншому випадку} \end{cases}$$
3. Оновлюємо стан годинника пристрою: $l_i = l'_i$, $c_i = c'_i$.
4. Подія маркується таймстампом $T_{local} = (l_i, c_i, \text{node\_id}_i)$.

#### Алгоритм 2: Отримання події з таймстампом $T_m = (l_m, c_m, \text{node\_id}_m)$

При отриманні події з іншого пристрою зчитується поточний фізичний час
$\text{pt}_i$:

1. Обчислюємо тимчасове значення фізичного компонента $l'_i$:
   $$l'_i = \max(l_i, \text{pt}_i, l_m)$$
2. Обчислюємо значення лічильника $c'_i$:
   $$c'_i = \begin{cases} \max(c_i, c_m) + 1, & \text{якщо } l'_i = l_i = l_m \\ c_i + 1, & \text{якщо } l'_i = l_i \\ c_m + 1, & \text{якщо } l'_i = l_m \\ 0, & \text{в іншому випадку} \end{cases}$$
3. Оновлюємо стан годинника пристрою: $l_i = l'_i$, $c_i = c'_i$.

Порівняння двох таймстампів $T_1 = (l_1, c_1, n_1)$ та $T_2 = (l_2, c_2, n_2)$
виконується лексикографічно:
$$T_1 < T_2 \iff (l_1 < l_2) \lor (l_1 = l_2 \land c_1 < c_2) \lor (l_1 = l_2 \land c_1 = c_2 \land n_1 < n_2)$$

### 3.2. JSON-Схема канонічної події (Event Schema)

Кожен івент записується в журнал як один рядок у форматі JSON (JSONL).

```json
{
  "$schema": "https://minddoc.md/schemas/event-v2.json",
  "hlc": "2026-07-02T12:00:00.000000Z-0007-macbook-sah",
  "actor": "sah@macbook-01",
  "op": "META_SET",
  "node": "7K3M9H2B5V8X1Z4Q6W9R3T1Y7U",
  "key": "status",
  "value": "in-progress",
  "prev": "85f3408a2a7a4f9a56e01a4e21a2f3a610287a419c8f1e2f3d4c5b6a7f8e9d0c",
  "sig": "ed25519(8fbc923a10...ef)"
}
```

Поле `prev` містить BLAKE3-хеш попереднього івенту цього конкретного журналу,
утворюючи криптографічний хеш-ланцюг (Tamper-evident log).

### 3.3. Структура файлової організації журналів

На диску користувача простір (Space) організований за наступною схемою:

```
space_alpha/
  journals/
    macbook-sah-01/
      2026-07.seg.jsonl.age      <-- Симетрично шифрований сегмент за липень 2026
      2026-07.seg.sig            <-- Ed25519 підпис цього сегмента
      HEAD                       <-- Поточний лічильник івенту та хеш останнього запису
  roster/
    roster.jsonl                 <-- Журнал конституції простору (ключі, права, девайси)
  views/                         <-- Матеріалізовані проекції (кеш, що регенерується)
    nodes/                       <-- Чисті людські .md файли
    semantic/                    <-- Символьні посилання для швидкого No-Read інференсу
  cache/
    index.sqlite                 <-- Локальна швидка база cr-sqlite
```

Сегменти закриваються наприкінці кожного місяця або при досягненні розміру 10
МБ. Закритий сегмент підписується приватним ключем девайса і стає абсолютно
імутабельним.

### 3.4. Семантика CRDT-злиття

Для злиття паралельних подій з різних журналів без конфліктів використовуються
наступні типи даних:

- **LWW-Register (Last-Write-Wins):** Використовується для метаданих ноди
  (назва, статус, власник). Пріоритет визначається за таймстампом HLC.
- **OR-Set (Observed-Remove Set):** Використовується для зв'язків графу. Кожне
  додавання або видалення ребра графу несе унікальний ідентифікатор події та
  HLC-мітку. Видалення перекриває лише ті додавання, які воно реально "бачило" у
  своїй причинно-наслідковій історії.
- **Grow-Only Sequence:** Використовується для текстових блоків та стрічки
  коментарів. Текст зшиваються за допомогою текстового CRDT-рушія (Yjs або
  Automerge), загорнутого в журнальні івенти.

## 4. Детермінована Матеріалізація та Компакція

### 4.1. Матеріалізація як чиста функція

Оскільки істина лежить виключно в журналах подій, генерація файлу документа на
диску є чистою математичною функцією від відсортованої за HLC множини івентів:

$$f(\text{node\_id}, \bigcup \text{Journals}) \to \text{content.md (canonical bytes)}$$

Вхідними даними є абсолютно всі івенти, що стосуються даної ноди. Оскільки
правила сортування HLC та правила CRDT-злиття є детермінованими, запуск цієї
функції на будь-якому пристрої призведе до генерації байт-в-байт ідентичного
Markdown-документа.

```typescript
// Концептуальний код матеріалізації нативного файлу
export function materializeNode(
  nodeId: string,
  events: Event[],
): MaterializedFile {
  const sortedEvents = events
    .filter((e) => e.node === nodeId)
    .sort((a, b) => compareHLC(a.hlc, b.hlc));

  let metadata: Record<string, any> = {};
  let textSegments: string[] = [];

  for (const event of sortedEvents) {
    switch (event.op) {
      case "META_SET":
        metadata[event.key] = event.value;
        break;
      case "BODY_APPEND":
        textSegments.push(event.value);
        break;
        // Інші оператори мутацій
    }
  }

  const yamlFrontmatter = yaml.stringify(metadata);
  const markdownBody = textSegments.join("\n");
  const bytes = Buffer.from(`---\n${yamlFrontmatter}---\n\n${markdownBody}`);

  return {
    bytes,
    sha256: crypto.hash(bytes),
  };
}
```

### 4.2. Робота з Локальним Кешем (cr-sqlite)

Для миттєвого рендерингу UI та побудови графів Tauri-додаток використовує
вбудовану базу даних SQLite з розширенням `cr-sqlite`.

- При завантаженні або синхронізації нових сегментів журналів, локальний
  демон-watcher витягує з них івенти і напряму закидає їх в таблиці `cr-sqlite`.
- База даних виступає виключно індексом швидкого доступу. Якщо базу даних
  видалено або пошкоджено, вона автоматично перезбирається з нульового стану за
  лічені секунди за допомогою перечитування файлів `.jsonl` з теки `journals/`.

### 4.3. Семантична Проекція "Zero IOPS"

Концепція "No-Read Ops" з v1 переноситься на рівень матеріалізованих видів.

- Замість того, щоб кодувати семантичні категорії в назвах реальних канонічних
  файлів, нічний цикл (див. Розділ 8) розраховує семантичний вектор документа і
  мапить його в один з 16 семантичних секторів.
- Локальний демон створює в директорії `views/semantic/[sector_hex]/` символьні
  посилання (symlinks) або легковантажні стаб-файли, що вказують на реальні
  документи в `views/nodes/`.
- Завдяки цьому будь-яка зовнішня утиліта або локальний ШІ може виконати швидке
  групування та сканування простору, просто прочитавши структуру теки
  `views/semantic/`, не відкриваючи і не зчитуючи вміст самих файлів.

### 4.4. Merkle-DAG самарі для дерева нод

Кожен вузол графу містить криптографічний Merkle-підпис, який обчислюється
рекурсивно:

$$H(\text{Node}) = \text{BLAKE3}\Big(\text{content\_hash} \parallel H(\text{Child}_1) \parallel H(\text{Child}_2) \dots \parallel H(\text{Child}_n)\Big)$$

Зміна будь-якої підзадачі або коментаря внизу дерева інвалідує хеші лише вгору
по ланцюгу до кореневої стратегічної ноди. Це дозволяє нічному циклу аналізувати
зміни з часовою складністю $O(\text{depth})$ замість повного перерахунку всього
піддерева $O(\text{subtree})$.

## 5. Криптографічний Контур та Ростер (Roster)

Безпека системи будується на припущенні, що транспортний канал (хмара) повністю
скомпрометований.

### 5.1. Трирівнева архітектура ключів

```
[ Device Key (Ed25519) ] ---> Підписує івенти та сегменти (локальний, приватний)
          |
          +---> [ Space Key (age, симетричний) ] ---> Шифрує файли для хмари
                     |
                     +---> [ Publish Key (AES) ] ---> Одноразовий ключ для вебу
```

- **Device Key (Ed25519):** Локальна пара ключів. Створюється на пристрої при
  першому запуску. Приватний ключ ніколи не залишає девайс, публічний —
  записується у Конституційний Журнал Ростера.
- **Space Key (симетричний ключ шифрування age):** Використовується для
  шифрування та дешифрування журнальних сегментів перед відправкою в хмару. Він
  обгортається (wrapped) окремо під публічний ключ кожного авторизованого
  девайса з Ростера.
- **Publish Keys (симетричний AES):** Одноразові ключі для публікації статичних
  снапшотів у веб через Cloudflare Worker.

### 5.2. Ростер (Roster) як Конституційний Журнал

Файл `roster/roster.jsonl` — це суверенний імутабельний журнал, який описує
склад учасників простору та їхні публічні ключі. Мутація ростера здійснюється
івентами `KEY_ADD`, `KEY_REVOKE`, `GRANT_CAPABILITY`.

**Сценарій: Реагування на викрадення ноутбука співробітника**

1. Директор зі свого робочого пристрою емітить у ростер івент `KEY_REVOKE` для
   публічного Ed25519-ключа викраденого ноутбука.
2. Локальний демон Директора генерує новий симетричний `Space Key (age)`.
3. Новий `Space Key` обгортається (шифрується) під публічні ключі всіх
   пристроїв, окрім викраденого.
4. Нові сегменти журналів шифруються новим `Space Key`. Викрадений пристрій
   більше не здатний розшифрувати нові надходження з хмари, хоча може бачити
   старі історичні дані, які він уже встиг викачати до моменту відкликання.

### 5.3. Специфікація Капабіліті-Грантів (Capability Grants)

Права доступу в системі є локальними даними, які підписуються криптографічно.
Жоден пристрій не прийме івент від іншого пристрою, якщо автор не має діючого
гранту на цю операцію.

```json
{
  "op": "GRANT",
  "to": "agent:claude-night@sah-macbook",
  "caps": ["node.propose", "link.propose", "summary.write"],
  "scope": "sector:marketing/**",
  "expires": "2026-10-01T00:00:00Z",
  "sig": "ed25519(sah@macbook-01)"
}
```

## 6. ШІ як Автономний Громадянин: Протокол PROPOSE/ACCEPT

Система відмовляється від парадигми ШІ як пасивного генератора тексту. ШІ-агенти
є повноцінними акторами з власними цифровими підписами та журналами подій.

```
[ ШІ-Агент (Claude) ]
        |
        v (Емісія івенту в свій журнал)
  { op: "PROPOSE", change: "Link(nodeA, nodeB)" }
        |
        v (Синхронізація через хмару)
[ UI користувача ] ---> Відображення пропозиції у черзі ACCEPT/REJECT
        |
        +---> Користувач тисне ACCEPT
        |
        v (Емісія івенту користувача)
  { op: "ACCEPT", target: "event_id_propose" } ---> Зв'язок стає канонічним
```

### 6.1. Двошвидкісний контур дій агента

- **Автономні дії (Fast Track):** Якщо агент має прямий капабіліті-грант
  (наприклад, `summary.write` для стиснення описів закритих задач), він
  безпосередньо емітить івент `UPDATE_META` у свій журнал. Зміна миттєво стає
  канонічною для всього простору.
- **Пропозиційні дії (Slow Track):** Якщо дія виходить за межі його прямих
  повноважень (наприклад, видалення завдання або створення стратегічного
  зв'язку), агент записує у свій журнал івент типу `PROPOSE`. Ця подія потрапляє
  в інтерфейсну чергу користувача на розгляд.

### 6.2. Репутаційний контур (Acceptance Rate)

Кожен девайс у просторі в процесі нічного циклу аналізує історію взаємодії з
кожним ШІ-агентом за останні 60 днів, вираховуючи показник репутації:

$$R_{agent} = \frac{\sum \text{ACCEPT}}{\sum \text{ACCEPT} + \sum \text{REJECT}}$$

Якщо $R_{agent} > 0.95$ для певного типу операцій (наприклад, лінкування
маркетингових задач), система автоматично пропонує Директору підвищити права
агента до автономии: _"Агент Claude-Night успішно поєднав 120 задач без жодного
відхилення. Бажаєте надати йому автономне право LINK_WRITE?"_.

### 6.3. Мультиагентність без какофонії

Коли в просторі працюють десятки агентів (персональні асистенти кожного
менеджера, нічний аналітик, збирач фінансових звітів), виникає ризик спаму
пропозиціями.

- Резолвер використовує пріоритетну чергу пропозицій, відсортовану за
  репутаційним показником $R_{agent}$ автора.
- Пропозиції від агентів з низьким рівнем репутації ($R_{agent} < 0.70$)
  автоматично згортаються в один щотижневий низькопріоритетний дайджест,
  повністю звільняючи фокус уваги людини.

## 7. Детерміновані Побічні Ефекти (Distributed Hooks)

Для запуску автоматичних процесів у зовнішньому світі (надсилання листа через
SMTP, виклик вебхука на Windmill, оновлення статусу в Jira) без використання
єдиного центрального сервера реалізовано алгоритм розподіленого консенсусу.

### 7.1. Детермінована елекція виконавця (Deterministic Election)

Коли в журналі з'являється івент, який потребує виконання зовнішньої дії
(`HOOK_TRIGGER`), кожен пристрій незалежно обчислює, хто з активних учасників
простору має виконати цю операцію.

Нехай $N_{online}$ — список ідентифікаторів пристроїв, від яких за останні 10
хвилин у хмарі з'явився хоча б один івент-heartbeat. Список сортується
лексикографічно.

Індекс виконавця $Idx$ обчислюється як:

$$Idx = \text{BLAKE3}(\text{hook\_id}) \pmod{|N_{online}|}$$

Усі пристрої приходять до абсолютно однакового результату. Обраний пристрій
$N_{online}[Idx]$ бере задачу на виконання.

### 7.2. Механізм оренди (Lease-holding) та ідемпотентність

1. Обраний пристрій миттєво емітить у свій журнал івент `HOOK_LEASE` з TTL
   (наприклад, 2 хвилини):
   `{ op: "HOOK_LEASE", hook_id: "X", expires: "2026-07-02T12:02:00Z" }`
2. Якщо за час TTL пристрій не записав івент `HOOK_SUCCESS` або `HOOK_FAILED`
   (наприклад, зник інтернет чи вимкнулося живлення), елекція повторюється по
   наступному колу:
   $$Idx_{fallback} = (\text{BLAKE3}(\text{hook\_id}) + 1) \pmod{|N_{online}|}$$
3. **Зовнішня ідемпотентність:** Будь-який запит до зовнішньої системи
   (Windmill/Jira) супроводжується унікальним ключем ідемпотентності:
   $$\text{idempotency\_key} = \text{BLAKE3}(\text{hook\_id} \parallel \text{payload})$$
   Навіть якщо через затримки синхронізації два пристрої одночасно смикнуть
   зовнішнє API, отримувач дедуплікує запит і не виконає дію двічі.

## 8. Нічний Цикл Консолідації та Когнітивний Шар

Вдиночі (або під час тривалих пауз) запускається когнітивний пайплайн
консолідації знань.

```
[ Старт нічного циклу ]
          |
          v (Пошук брудних шляхів через Merkle-DAG)
[ Зчитування івент-логів та дельт за день ]
          |
          v (Локальний SLM: генерація Node Summary та RAG-індексів)
[ Оновлення семантичних ембедінгів та шардів Zero-IOPS ]
          |
          v (Велика LLM: генерація PROPOSE-івентів у журнал Claude)
[ Пошук прихованих зв'язків та семантичний аудит конфліктів ]
          |
          v
[ Компакція та архівація старих сегментів у .cold/ ]
```

### 8.1. Інкрементний RAG

Кожна нода обвешивається трьома динамічними RAG-індексами, які зберігаються у
локальному індексі SQLite:

1. **Node Summary:** Семантичний витяг з канонічного вмісту самої ноди.
2. **Upward Context (Ancestor RAG):** Накопичений векторний контекст усіх
   батьківських нод вгору по ієрархії графу. Дає розуміння глобальної стратегії.
3. **Downward Context (Descendant RAG):** Контекст підзадач, коментарів та
   операційних регламентів, що лежать нижче.

Завдяки Merkle-DAG структурі, якщо за день змінився лише один документ, система
перераховує embeddings тільки для цієї ноди та її прямих предків до кореня.
Часова складність оновлення RAG-бази становить $O(\text{depth})$ замість повного
перерахунку графу.

### 8.2. Семантичний аудит та виявлення прихованих зв'язків

Велика модель (наприклад, Claude 3.5 Sonnet через API) аналізує денні зміни і
робить інтелектуальний аудит:

- Шукає логічні протиріччя в дедлайнах та оцінках між різними командами.
- Знаходить приховані дублікати завдань у різних департаментах.
- Емітить у свій журнал івенти `PROPOSE` на лінкування суміжних контекстів.

### 8.3. Аналітика здоров'я системи (Operational Health Signals)

Замість езотеричного "передбачення вигорання" нічний цикл вираховує реальні
показники семантичної ентропії та стійкості команди:

1. **Частка нод-сиріт (Orphan Rate):** Кількість створених задач, які не мають
   жодного логічного зв'язку з глобальними стратегічними цілями компанії.
2. **Тренд PROPOSE-відхилень (Rejection Trend):** Показник того, наскільки часто
   користувачі відхиляють пропозиції ШІ-агентів у конкретному секторі. Якщо
   показник зростає, це індикатор того, що контекст сектора занадто хаотичний і
   моделі втрачають розуміння процесів.
3. **Латентність статусних переходів:** Реальний час перебування завдань у стані
   "in-progress" до моменту закриття, що виявляє приховані блокери процесів.

## 9. Продуктовий Стек та Нативна Реалізація (MVP)

### 9.1. Tauri як архітектурний контейнер

Продукт поставляється як єдиний інсталятор під Windows, macOS та Linux.

- **Rust Core (Backend):** Реалізує watcher файлової системи, локальний
  криптографічний контур (Ed25519, `age` шифрування), HLC-годинники, локальну
  базу `cr-sqlite` та CLI-інтерфейс.
- **TypeScript (Frontend):** Легковажний інтерфейс (React/Vue/Svelte), який
  рендериться всередині системного Webview. Спілкується з Rust-ядром через
  безпечні біндінги IPC (Inter-Process Communication).
- **Мобільна версія:** Створюється за допомогою Tauri Mobile (Rust-ядро
  переноситься на iOS/Android як нативна бібліотека), забезпечуючи локальний
  резолвінг без зміни логіки.

### 9.2. Onboarding та покроковий план розробки

#### Етап 1: Однокористувацьке Ядро (Phase 1)

Реалізація локального журналу подій, HLC, SQLite-індексу та матеріалізації
чистих `.md` файлів на диску користувача. Синхронізація через Google Drive
одного користувача між двома його пристроями.

#### Етап 2: Конституція Простору та Ростер (Phase 2)

Додавання крипто-контуру. Створення `roster.jsonl`, генерація Ed25519 ключів
пристроїв, симетричне шифрування сегментів журналів для хмари за допомогою
`age`. Додавання другого користувача в простір.

#### Етап 3: ШІ-Громадянин та PROPOSE-черга (Phase 3)

Інтеграція першого ШІ-агента зі своїм підписом і журналом. Реалізація інтерфейсу
черги затвердження пропозицій ШІ у Tauri-додатку.

#### Етап 4: Розподілені Хуки та Веб-шлюз (Phase 4)

Реалізація алгоритму детермінованої елекції виконавців для хуків, підключення
Windmill-шлюзу. Запуск Cloudflare Worker для публікації безпечних шифрованих
снапшотів простору у веб.
