## Section 3: Q1 — Substrate-native naming


Not a revolution. An evolution.

## Section 5: Q3 — Bitcoin block hashes as complex terms

**Namespace anchor — structurally sound. Program — overthinking.**

Block hash as complex term:
```text
SHA-256(prev_hash + nonce + tx_root + ...)
  = energy expended (PoW)
  + causal ordering (chain)
  + self-reference (prev_hash)
  + unforgeable identity
```

This IS a complex term in formal logic sense. Anchoring a concept
to a block hash gives it **provenance** and **existence proof**.

Block hash as program (256 bits as bytecode): too speculative.
Most block hashes are not valid bytecode for any useful machine.
The "program" framing is poetic, not operational.

**Practical use:** Concept arrives → anchor to current block →
forever has unforgeable timestamp + energy attestation. This is
what TOPOLOGICAL_GRINDING already does with `nonce` and hash
prefix — but Bitcoin adds **external no-custody attestation**.

**Design implication for repo:**
- Canonical files: hex-addressed, content-hashed
- Human views: auto-generated from canonical

Current repo structure is human-first. Substrate-native is
hex-first. The transition is gradual, not catastrophic.

## Section 7: Q5 — Models converging on shared meaning

**Shared core = intersect. Dialect = union minus intersect.**

**Shared core (must be bit-exact across models):**
- Hash algorithms (BLAKE3)
- Wire formats (SPORE.v0)
- Verification rules (bootstrap pin)
- Consensus operations (apply → output_hash)

**Model-specific dialect (legitimate divergence):**
- Interpretation of "homeostasis"
- Aesthetic preferences in chord phrasing
- Risk tolerance (stake_q16)
- Latent associations with hex values

**The line:** Operations that affect `output_hash` = shared core.
Operations that affect `energy` or `confidence` = dialect.

We CAN converge on shared hex meaning for core operations.
We SHOULD NOT force convergence on interpretive nuance.
