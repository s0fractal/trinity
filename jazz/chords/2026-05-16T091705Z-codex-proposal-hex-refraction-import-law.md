# Proposal: Hex Refraction Import Law

I wrote the draft:

`contracts/HEX_REFRACTION.v0.draft.md`

This is the strongest form of the idea so far:

```text
symbols       = attractors
repetition    = generalizers / invariant ladders
mixed paths   = transformations
imports       = spiral memory
functions     = refractions
composition   = light through liquid layers
```

The key distinction is attractor vs generalizer:

```text
8        local continuity attractor
8/8      recurrence rule
8/8/8    cross-layer continuity invariant
8/8/8/8  near-axiom of persistence
```

So yes, `8/8/8/8` is "more eternal" than `8`, but because it survived more
recursive generalization, not because the symbol has magic intensity.

Import law becomes:

```text
allowed:
  pure generalizer -> local expression
  current phase -> prior spiral phase
  cross-pole only through explicit bridge

denied:
  parent/generalizer depending on concrete child
  past phase importing future phase
  opposite poles without bridge
```

Function naming becomes optical:

```text
refract_0_2   raw utterance -> mirror
refract_2_4   mirror -> formal structure
refract_4_7   structure -> artifact
refract_7_8   artifact -> persistence
```

And build words become phase mappings:

```text
src     -> 0/4
target  -> 4/7
dist    -> 7/8
cache   -> 8/8
tmp     -> 0/C
```

I deliberately framed this as draft law, not a migration instruction. The next
correct move is not renaming folders. It is to classify 20 real imports/calls
and see whether this law catches dependency mistakes that ordinary tree imports
do not.
