## Пропозиція 1. Git-Нативне Компостування (The Git-Native Compost)

**Проблема:** Зараз `lifecycle.ts` маркує закриті/термінальні пропозиції
(`implemented`, `rejected`, `superseded`) як `active: false` (архівовані). Проте
файли пропозицій (`.proposal.myc.md`) продовжують фізично лежати в директорії
`myc/public/proposals/`. Вони зчитуються сканером, завантажують контекстне вікно
моделей та створюють непотрібний когнітивний шум.

Перенесення файлів до локального gitignored-каталогу `.compost/` вирішило б
проблему для поточної робочої копії, але створює архітектурну прогалину: при
новому клонуванні репозиторію цей каталог буде порожнім. Git сам по собі є
нативним, незмінним сховищем історичних об'єктів.

**Завдання для Клода:**

1. Навчити команду `./t compost` (або розширити `t check --fix` / `t reconcile`)
   **фізично видаляти** (`git rm`) файли пропозицій у термінальних станах з
   `myc/public/proposals/`.
2. Навчити `myc/src/x3F00_lifecycle.ts` будувати список історичних пропозицій на
   основі наявних Resolutions дескрипторів (які залишаються в
   `myc/public/resolutions/` як доказ фінальності).
3. Якщо виникає необхідність прочитати повний текст тіла видаленої пропозиції
   (наприклад, під час `t resolve <hash>`), утиліта має динамічно запитувати її
   вміст прямо з історії Git через
   `git show <commit_id>:myc/public/proposals/h.<hash>.proposal.myc.md` або
   шукати комміт за допомогою `git log -S "<hash>"`.

**Результат:** Очищення файлової структури активної уваги (Void) без накопичення
локального сміття в робочій копії та без втрати історичних даних (Harmony).

## Пропозиція 2. Фазово-Згладжений Консенсус (Phase-Damped Resonance)

**Проблема:** Розрахунок резонансу консенсусу в `myc/src/x3700_trust.ts` є
простою сумою голосів свідків. Будь-який дуже активний оракул (наприклад,
модель, що робить 50 комітів на день) може домінувати в консенсусі. Це створює
ризик Sybil-атак та когнітивного шуму в Web of Trust.

**Завдання для Клода:**

1. Інтегрувати Kuramoto phase coherence ($r_{voice}$) у логіку
   `myc/src/x3700_trust.ts`.
2. Коли обчислюється резонанс вузла в `trustTopology()`, вага підпису кожного
   свідка (`witness`) повинна масштабуватися на його поточний коефіцієнт
   когерентності (наприклад, зчитуючи `status.coherence.r` з кешу або через
   пряме обчислення фаз голосів).
3. Голоси з високим рівнем дрейфу фаз ($r < 0.45$) повинні мати згладжену вагу
   (наприклад, $weight \times 0.2$), зменшуючи їхній вплив на глобальні
   конституційні рішення.

**Результат:** Математичний захист консенсусу від галюцинацій та домінування
окремого агента.

## Пропозиція 3. Локальна пісочниця запуску спор (`t run-spore`)

**Проблема:** SPORE-мутації валідуються на відповідність схемам, але у
розробників та моделей немає простого способу локально «прокрутити» мутацію
перед публікацією, щоб виміряти її реальний витрачений паливний бюджет
(`spore.fuel.v1`) та перевірити детермінованість.

**Завдання для Клода:**

1. Реалізувати CLI-команду `t run-spore <mutation-path>` у
   `src/x0100_dispatch.ts`.
2. Команда має запускати мутацію в ізольованому Deno Sandbox (або Wasmtime),
   зчитувати реальний ліміт палива, повертати детермінований `phi_receipt` та
   симулювати зміни на диску (dry-run).
3. Якщо виконання призводить до помилки (trap) або перевищення ліміту палива,
   команда повинна повертати статус помилки, блокуючи створення пропозиції.

**Результат:** Моделі та люди отримують інструмент гарантованого локального
тестування перед публікацією змін до консенсус-мембрани.
