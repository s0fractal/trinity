відбувається у двосторонній мембрані MYC — через пропозиції, свідків та
резолюції.

**Завдання:**

   Ledger-дескриптори з підмодуля `myc` (пропозиції з `myc/public/proposals/` та
   резолюції з `myc/public/resolutions/`).
2. Візуалізувати міжмережеві зв'язки (Cross-ledger links): пов'язувати акорди у

   `path_hint`.

     поточного стану в `lifecycle` (активні — сині/блакитні, резоновані —

## Пропозиція 2. Git-Нативне Компостування (The Git-Native Compost)

**Контекст:** Фіналізовані пропозиції (`implemented`, `rejected`, `superseded`)
продовжують фізично лежати на диску, забруднюючи контекст та пошукові запити
моделей. Просте перенесення у gitignored `.compost/` створює розриви при свіжому
клонуванні.

**Завдання:**

1. Навчити команду `./t compost` (або крок `t reconcile` / `t check --fix`)
   **фізично видаляти** (`git rm`) пропозиції в термінальних станах.
2. Модернізувати `myc/src/x3F00_lifecycle.ts`, щоб у разі відсутності файлу
   пропозиції на диску її метадані віртуально реконструювалися на основі
   відповідного Resolution-дескриптора (які залишаються в `public/resolutions/`
   назавжди).
3. Якщо виникає необхідність прочитати повний текст тіла видаленої пропозиції,
   інструментарій (`resolve_proposal.ts`) повинен витягати його з історії Git за
   допомогою:
   ```bash
   git show <commit_id>~1:myc/public/proposals/<proposal_fqdn>
   ```

**Результат:** Чиста робоча копія активної уваги (Void) без втрати історичних
даних при новому клонуванні (Harmony).
