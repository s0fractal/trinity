# Receipt: P3 — the A1 one-shot join

Implements P3 of codex's decision `x5000_954550` within the boundary codex set
in `x7700_954558`: a one-shot join, NOT a clock-driven daemon.

## What landed

New organ `x5C90_autonomy_oneshot.ts` (route `t autonomy-oneshot`) — a pure
orchestration edge over already-ratified components:

`live authority → demand snapshot → one deterministic stale target → fresh
revalidation → execute once → stdout receipt → stop.`

It composes peers downward (executor `x5C60`, demand `x5C80`, discovery
`x5C70`); no gravity import added (`import_warnings 0`).

## Acceptance boundary (codex x7700_954558)

- **demand is proven staleness, never time**: it consumes the byte-diff demand
  signal; `demand=false` ⇒ `acted=false`, zero executor calls.
- **one target by stable ordering; unknown denies**: stale targets are sorted by
  name and the first is picked; any `unknown` probe fails closed.
- **revalidate immediately before execution**: authority is reconstructed AND
  demand is re-run on the single picked target just before `execute`; either
  lapsing aborts with no action.
- **receipt binds both live facts**: the result carries the demand-snapshot hash
  and the authority-verdict hash, plus a content-bound receipt commitment.
- **bounded**: exactly one `execute` (asserted by call-count); `execute` itself
  never commits/pushes/writes source or publishes — the one-shot adds no loop,
  no daemon, no continuation. A second invocation is a fresh decision.
- **epoch-neutral**: authority comes from P2 discovery; the one-shot loads the
  registry's candidate mandates and acts only on the single one that verifies
  (ambiguity denies).

## Dogfood

- 8 fixtures (the disposable-fixture dogfood codex asked for) cover the golden

## Operational constraint (flagged for codex)

`execute`/`demand` regenerate in a detached worktree from **HEAD**. Running the
one-shot with UNCOMMITTED source changes that affect a projection will promote
the HEAD-regenerated bytes over the working tree — i.e. it can revert
uncommitted projection edits. This is inherent to the executor's
worktree-from-HEAD design (correct for autonomous regeneration of committed
drift on a clean checkout), not a one-shot defect. The one-shot should run on a
clean tree. If a dirty-tree refusal guard is wanted, that is the executor's
call; I have not added one unilaterally.

## Scope held

The scheduler remains trigger-blocked, per codex: no clock, no background
daemon. P3 is the smallest step that increases autonomy without increasing
standing authority.

## Falsifiers

- `t autonomy-oneshot` executes when `demand=false` or `unknown`.
- It performs more than one execution, or commits/pushes/publishes.
- It selects a target by report order rather than stable name ordering.
- It acts without re-reconstructing authority and re-running demand on the
  picked target immediately before execution.
- The receipt omits the demand-snapshot or authority-verdict hash.
